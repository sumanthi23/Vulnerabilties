Learning guide

This app has two SSRF-vulnerable features, each modeling something you'll actually run into hunting for real bugs: a filtered fetch (most real targets have some check) and a blind fetch (most real bugs don't hand you a response body). Three objectives.

{{ '✓' if flag1 else '1' }}
Bypass the whitelist, reach the internal admin panel
The import feature only accepts URLs containing cdn.shopstock.local. That check runs against the raw string, not the parsed host — the same class of bug behind real whitelist-bypass reports on HackerOne/Bugcrowd.
Hint 1

The filter checks whether the trusted string appears anywhere in the URL. A URL can contain a hostname without the request actually going there.

Hint 2 (userinfo trick)

Try:
http://cdn.shopstock.local@127.0.0.1:5001/admin/secret
Everything before @ is userinfo, not the host — browsers and HTTP clients ignore it for routing, but a naive string check doesn't know that.

Hint 3 (fragment trick)

Also works:
http://127.0.0.1:5001/admin/secret#cdn.shopstock.local
Everything after # is a fragment — never even sent to the server — but it's still part of the string the filter sees.

{{ '✓' if flag2 else '2' }}
Steal cloud credentials and watch them get reused
Same bypass, different destination: the fake cloud metadata endpoint. This is the pattern behind the 2019 Capital One breach and CVE-2025-53767 (Azure OpenAI).
Hint 1 (payload)

Try:
http://cdn.shopstock.local@127.0.0.1:5001/latest/meta-data/iam/security-credentials/ISRM-WAF-Role

Hint 2 (what to expect)

You'll see an access key + secret key, then a second block showing customer records — this app automatically replaying the stolen credentials against a separate service.

{{ '✓' if flag3 else '3' }}
Blind SSRF: port-scan the internal network by timing
The webhook tester below (no whitelist at all) never shows you a response — only whether the request succeeded and how long it took. That's still enough to map what's alive internally, a real technique used when a target's SSRF is blind.
Hint 1

Try the webhook tester against http://127.0.0.1:5001/ (open — the internal service) and then against http://127.0.0.1:5999/ (closed). Compare the result and timing.

Hint 2

An open port that isn't listening HTTP still often responds instantly with "connection refused" while a closed/filtered one hangs to your timeout. Fast rejection vs. a timeout is the signal — you're mapping the network without ever reading a response body.

Bug bounty toolkit

Notes for taking this from "I can trigger SSRF in a lab" to "I can find it in a real program."

Parameters worth fuzzing on real targets

Any feature that fetches something server-side tends to hide behind params like: url, uri, path, dest, redirect, continue, next, return, callback, domain, host, site, reference, data, feed, view, page, out, to, val.

Confirming blind SSRF on a real target

Point the suspect parameter at a URL you control on an out-of-band listener — webhook.site, Burp Collaborator, or Interactsh — and watch for the callback. No callback doesn't always mean "safe": some targets block egress but still resolve DNS, so a DNS-only interaction can still confirm the bug.

Bypass techniques worth trying against a filter

Userinfo (trusted@evil.com), fragment (evil.com#trusted), DNS hierarchy (trusted.evil.com), URL-encoding (%40 for @), decimal/octal/hex IP forms for 127.0.0.1, and — if the trusted host itself has an open redirect — chaining through that redirect to land anywhere.

What actually makes SSRF worth reporting

Reaching 127.0.0.1 alone is usually a low/info-severity finding. Severity jumps when it reaches something with real impact behind it: cloud metadata credentials, an internal admin API, or a service with no auth because "only internal callers can reach it." Always show the full chain in your report, not just the initial fetch.
